# 实盘交易架构概述

## 核心概念

实盘交易架构（Live Trading Architecture）是指将量化策略从回测环境迁移到真实市场的完整技术栈。一个稳健的实盘系统需要解决**低延迟、高可用、可扩展**三大核心挑战。相较于回测环境，实盘交易面临订单执行不确定性、市场微观结构噪声、服务器故障等真实风险，任何一个环节的疏漏都可能导致重大损失。

实盘交易架构通常由四大核心模块组成：**数据层**负责实时行情接入与清洗；**策略层**负责信号生成与投资决策；**执行层**负责订单路由与券商接口对接；**风控层**负责实时仓位监控与风险限制检查。四层之间通过消息队列或内存共享进行高效通信，整体设计遵循**面向失败设计（Design for Failure）**原则，确保单点故障不会导致系统崩溃。

对于目标CMU/MIT量化金融硕士项目的学生而言，理解实盘架构不仅是工程能力的体现，更是量化研究的核心竞争力。顶级量化机构（如Citadel、Two Sigma、DE Shaw）在面试中经常考察候选人对交易系统架构的理解深度。

## 方法论

### 架构设计原则

实盘交易系统的设计需要遵循以下核心原则：

1. **解耦与模块化**：各模块之间通过标准化接口通信，策略层无需关心底层执行细节。推荐使用事件驱动架构（Event-Driven Architecture），以消息队列为核心实现模块间解耦。

2. **低延迟优先**：对于高频策略，延迟以微秒（μs）计算。系统设计需考虑：
   - 内存池（Memory Pool）避免动态分配
   - 避免锁竞争，使用无锁数据结构
   - 批处理与向量化计算
   - 专用硬件加速（FPGA/网卡）

3. **高可用设计**：
   - 主备切换机制（Active-Standby）
   - 断路器（Circuit Breaker）防止雪崩
   - 异步日志与监控，不阻塞主交易路径
   - 优雅降级（Graceful Degradation）

### 主流技术栈

| 组件 | 推荐技术 | 适用场景 |
|------|---------|---------|
| 行情接入 | QuickFIX, WebSocket | 实时行情 |
| 策略引擎 | C++, Java, Python | 策略执行 |
| 消息队列 | Kafka, ZeroMQ | 进程间通信 |
| 数据库 | TimescaleDB, InfluxDB | 时序数据存储 |
| 风控系统 | 自研/AlgoSec | 合规监控 |

## 代码实现

以下是一个简化的事件驱动交易架构示例（Python）：

```python
import asyncio
from dataclasses import dataclass
from typing import Dict, Callable, Optional
from enum import Enum
import logging

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

class EventType(Enum):
    MARKET_DATA = "market_data"
    SIGNAL = "signal"
    ORDER = "order"
    FILL = "fill"
    RISK_CHECK = "risk_check"

@dataclass
class Event:
    event_type: EventType
    timestamp: float
    payload: Dict
    source: str

class EventBus:
    """事件总线：模块间通信的核心枢纽"""
    def __init__(self):
        self._subscribers: Dict[EventType, list] = {e: [] for e in EventType}
        self._event_queue: asyncio.Queue = asyncio.Queue()
    
    def subscribe(self, event_type: EventType, callback: Callable):
        self._subscribers[event_type].append(callback)
    
    async def publish(self, event: Event):
        await self._event_queue.put(event)
        logger.info(f"Event published: {event.event_type.value}")
    
    async def process_events(self):
        while True:
            event = await self._event_queue.get()
            for callback in self._subscribers[event.event_type]:
                try:
                    await callback(event)
                except Exception as e:
                    logger.error(f"Callback error: {e}")

class StrategyEngine:
    """策略引擎：信号生成"""
    def __init__(self, event_bus: EventBus, strategy_name: str):
        self.event_bus = event_bus
        self.strategy_name = strategy_name
        self.position = 0
        
    async def on_market_data(self, event: Event):
        price = event.payload.get('last_price')
        # 简化策略：简单移动平均交叉
        signal = self._generate_signal(price)
        if signal != 0:
            await self.event_bus.publish(Event(
                EventType.SIGNAL,
                timestamp=event.timestamp,
                payload={'signal': signal, 'price': price, 'strategy': self.strategy_name},
                source=self.strategy_name
            ))
    
    def _generate_signal(self, price: float) -> int:
        # 伪代码：实际需要维护历史数据
        return 1 if price > 100 else -1

class RiskManager:
    """风控模块：交易前检查与交易后监控"""
    def __init__(self, event_bus: EventBus):
        self.event_bus = event_bus
        self.max_position = 1000
        self.max_loss_pct = 0.02
        self.current_pnl = 0.0
        
    async def pre_trade_check(self, event: Event) -> bool:
        signal = event.payload.get('signal', 0)
        position = event.payload.get('current_position', 0)
        
        # 检查仓位限制
        if abs(position + signal) > self.max_position:
            logger.warning(f"Position limit exceeded: {position + signal}")
            return False
        
        # 检查亏损限制
        if self.current_pnl < -self.max_loss_pct:
            logger.error("Daily loss limit exceeded, halting trading")
            return False
        
        return True
    
    def update_pnl(self, pnl: float):
        self.current_pnl = pnl

class ExecutionHandler:
    """执行层：订单发送与成交回报处理"""
    def __init__(self, event_bus: EventBus):
        self.event_bus = event_bus
        self.broker = None  # 实际连接券商API
        
    async def send_order(self, signal_event: Event):
        signal = signal_event.payload.get('signal')
        price = signal_event.payload.get('price')
        
        # 模拟订单
        order = {
            'symbol': 'SPY',
            'side': 'BUY' if signal > 0 else 'SELL',
            'quantity': abs(signal),
            'order_type': 'LIMIT',
            'price': price
        }
        
        logger.info(f"Order sent: {order}")
        # 实际场景：调用broker API
        # response = await self.broker.send_order(order)
```

## 应用与局限

### 实际应用场景

1. **多策略并行**：实盘架构支持同时运行多个策略，每个策略独立订阅行情、生成信号，通过风控层统一管理仓位。

2. **跨市场交易**：支持A股、港股、美股等多市场同时交易，需要解决时区差异、市场规则不一致等技术挑战。

3. **组合管理**：大型量化基金通常采用Portfolio级别的风控与优化，实盘架构需要支持组合层面的信号聚合与仓位分配。

### 局限性

1. **回测-实盘差距（Backtest-Trading Discrepancy）**：实盘中的滑点、流动性限制、执行延迟等因素使得实盘表现往往不如回测。这一问题被称为**Paper-Live Gap**，是量化交易的核心难题之一。

2. **技术复杂度高**：构建一套生产级别的实盘系统需要深入的工程能力，对于个人投资者而言门槛较高。可以考虑使用托管服务（如Interactive Brokers的Client Portal API、聚宽、米筐等平台）降低技术门槛。

3. **数据质量依赖**：实盘系统的表现高度依赖数据质量，行情数据中的噪声、缺失、乱序都需要完善的处理机制。
