## 4.1 交易API与系统架构

### 主流API对比

| 接口 | 语言 | 特点 | 适合人群 |
|------|------|------|----------|
| Interactive Brokers (IBAPI) | Python/C++/Java | 品种全、费用低、文档好 | 入门实盘首选 |
| Alpaca | Python | 免费、RESTful、简洁 | 美股数字货币 |
| Binance API | Python | 数字货币、延迟低 | 加密货币策略 |
| ZeroMQ/FPGA | C++ | 超低延迟 | 做市商/高频 |

### 系统架构分层

```
信号生成层（策略）
    ↓
订单管理层（OMS）—— 订单验证、优先级、拆分
    ↓
交易执行层（EMS）—— API调用、订单路由
    ↓
风控层（Risk）—— 仓位限制、止损检查
    ↓
券商/交易所
```

### 核心组件

- **订单管理OMS**：管理挂单/撤单、追踪订单状态
- **执行管理系统EMS**：负责实际与券商API交互
- **风控引擎**：前置检查（仓位、流动性、风险限额）
- **数据订阅**：实时行情feed（LV2行情、逐笔成交）

## 4.2 订单类型与指令

### 基本订单

- **市价单（Market Order）**：立即以最优价格成交，适合流动性好的品种
- **限价单（Limit Order）**：指定价格成交，防止价格滑移
- **止损单（Stop Order）**：价格触发后转为市价单，用于止损
- **止损限价单（Stop-Limit）**：触发后以限价单执行

### 高级订单

- **追踪止损（Trailing Stop）**：价格朝有利方向变动时自动调整止损价
- **冰山订单（Iceberg）**：大单拆成小单，只显示部分量
- **时间加权执行（TWAP/VWAP）**：大单分时段执行，跟踪市场价

## 4.3 滑点与佣金建模

### 滑点（Slippage）

实盘成交价与预期价格的偏差，来源于：
- 市场冲击（大单影响市场价格）
- 流动性不足
- 订单执行延迟

**滑点估算经验值：**
- 大盘股（苹果、微软）：< 0.05%
- 中盘股：0.05% ~ 0.15%
- 小盘股/数字货币：0.15% ~ 0.5%

**建模公式：**
```
实际成交价 = 报价 × (1 ± slippage_rate)
滑点成本 = |实际成交价 - 报价| × 成交量
```

### 佣金结构

| 券商 | 股票 | 期权 | 期货 |
|------|------|------|------|
| IBKR | $0.005/股 | $0.65/合约 | $0.85/合约 |
| Alpaca | $0/股 | - | - |
| Interactive Brokers | 最低$0.35/笔 | | |

**年化交易成本估算：**
```
年化成本 = (佣金 + 滑点) × 年交易次数 / 平均资金
```

## 4.4 资金管理框架

### 仓位计算（Kelly Criterion基础）

```python
import numpy as np

def kelly_fraction(win_rate, avg_win, avg_loss):
    """
    Kelly公式：f* = (bp - q) / b
    b: 赔率（avg_win / avg_loss）
    p: 胜率
    q: 败率 (1-p)
    """
    b = avg_win / avg_loss
    p = win_rate
    q = 1 - p
    f = (b * p - q) / b
    return max(0, min(f, 0.25))  # 上限25%避免过度集中

def position_size(capital, price, stop_loss_pct, kelly_f=None):
    """
    按止损比例计算仓位
    capital: 资金
    price: 买入价格
    stop_loss_pct: 止损比例（如0.02表示2%）
    """
    if kelly_f is None:
        kelly_f = 0.1  # 默认用10%仓位
    risk_amount = capital * kelly_f * stop_loss_pct
    shares = int(risk_amount / (price * stop_loss_pct))
    return shares
```

### 风险平价策略

```python
def risk_parity_weights(volatility_dict):
    """
    volatility_dict: {asset: annual_volatility}
    风险平价：每个资产对组合总风险的贡献相等
    """
    inv_vol = {k: 1/v for k, v in volatility_dict.items()}
    total = sum(inv_vol.values())
    return {k: v/total for k, v in inv_vol.items()}
```

## 4.5 风险管理核心指标

### 仓位限制规则

```python
class RiskManager:
    def __init__(self,
                 max_position_pct=0.05,     # 单品种最大仓位5%
                 max_sector_pct=0.20,       # 单行业最大仓位20%
                 max_leverage=1.0,          # 最大杠杆
                 max_drawdown_pct=0.15):    # 最大回撤阈值
        self.max_position_pct = max_position_pct
        self.max_sector_pct = max_sector_pct
        self.max_leverage = max_leverage
        self.max_drawdown_pct = max_drawdown_pct

    def check_order(self, order, portfolio, market_data):
        """下单前风控检查"""
        errors = []

        # 1. 仓位上限检查
        new_position = portfolio.get(order.symbol, 0)
        position_pct = new_position * market_data[order.symbol].price / portfolio.total_value
        if position_pct > self.max_position_pct:
            errors.append(f"仓位超限: {position_pct:.2%} > {self.max_position_pct:.2%}")

        # 2. 杠杆检查
        if portfolio.total_exposure / portfolio.total_value > self.max_leverage:
            errors.append("杠杆超限")

        # 3. 流动性检查
        daily_volume = market_data[order.symbol].volume
        if order.quantity > daily_volume * 0.01:  # 超过成交量1%
            errors.append("流动性不足：大单可能冲击市场")

        return errors

    def check_drawdown(self, portfolio, peak_value):
        """回撤检查，超过阈值触发策略停止"""
        current_drawdown = (peak_value - portfolio.total_value) / peak_value
        if current_drawdown > self.max_drawdown_pct:
            return "REBALANCE"  # 降仓或止损
        return "OK"
```

### VaR（Value at Risk）

```python
import numpy as np

def historical_var(returns, confidence=0.95, horizon=1):
    """
    Historical VaR: 基于历史收益率分布的分位数
    returns: 日收益率序列
    confidence: 置信度（如0.95）
    horizon: 持有天数
    """
    var = np.percentile(returns, (1 - confidence) * 100)
    return var * np.sqrt(horizon)  # 方差缩放
```

## 4.6 回测到实盘的Gap

| 问题 | 回测 | 实盘 |
|------|------|------|
| 成交价格 | 收盘价/成交价 | 取决于订单执行时市场 |
| 滑点 | 假设固定% | 动态变化 |
| 流动性 | 无限量成交 | 大单可能无法成交 |
| 信号延迟 | 无延迟 | 网络/处理延迟 |
| 未来函数 | 常见错误 | 不存在 |
| 手续费 | 固定比例 | 阶梯费率 |

### 解决方案

1. **Paper Trading（模拟交易）**：先在模拟盘跑3个月
2. **小资金实盘**：实盘金额=回测金额×5%，验证执行逻辑
3. **回测优化**：加入滑点模型、市场冲击模型、手续费敏感性分析
4. **事件驱动回测**：tick级数据，避免未来函数

## 4.7 常见陷阱

- **生存者偏差（Survivorship Bias）**：只使用当前存活股票回测，忽略退市股票
- **前视偏差（Look-ahead Bias）**：使用回测时无法获得的信息
- **过度拟合（Overfitting）**：策略在历史数据上表现优异但实盘失效
- **流动性陷阱**：小市值股票回测容易，实盘难以成交
- **交易成本低估**：实际佣金和滑点远高于回测假设
